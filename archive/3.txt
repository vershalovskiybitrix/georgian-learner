Plan: Georgian Language Learning Browser Extension
Context
Пользователь хочет браузерный плагин для пассивного изучения грузинского алфавита. Идея: в процессе обычного чтения часть латинских/кириллических букв заменяется на соответствующие грузинские — мозг постепенно привыкает к новым знакам без сознательного усилия.

V1.0 — транслитерация. V2.0 — добавить перевод целых слов (архитектура должна это учитывать).

Требования по итогам уточнений
Исходные алфавиты: кириллица и латиница одновременно
Визуальная подсветка: да, переключается в настройках
Таблица соответствий: открывается как страница настроек расширения (options.html)
Динамический контент: MutationObserver включён по умолчанию, но с переключателем в popup
Технологии
Слой	Решение	Почему
Platform	WebExtension API / Manifest V3	Единый стандарт Chrome + Vivaldi, требование Chrome Web Store
Language	Vanilla JS (ES2020 modules)	Нет сборщика = меньше точек отказа, легче аудит безопасности
Storage	chrome.storage.sync	Настройки синхронизируются между устройствами
DOM	TreeWalker + MutationObserver	Нативный API, нет зависимостей, высокая производительность
Mappings	JSON-файлы в /data/mappings/	Читаемо, редактируемо без кода, расширяемо на другие языки

Формат таблицы соответствий (JSON)
{
  "version": "1.0",
  "source": "latin",
  "target": "georgian",
  "description": "Latin to Georgian phonetic mapping",
  "mappings": [
    { "from": "kh", "to": "ხ", "priority": 10, "note": "aspirated k" },
    { "from": "ts", "to": "ც", "priority": 10 },
    { "from": "ch", "to": "ჩ", "priority": 10 },
    { "from": "sh", "to": "შ", "priority": 10 },
    { "from": "zh", "to": "ჟ", "priority": 10 },
    { "from": "gh", "to": "ღ", "priority": 10 },
    { "from": "dz", "to": "ძ", "priority": 10 },
    { "from": "a",  "to": "ა", "priority": 1 },
    { "from": "b",  "to": "ბ", "priority": 1 }
  ]
}
Важно: priority определяет порядок разбора — мультисимвольные паттерны (kh, ts) имеют приоритет над односимвольными. Алгоритм всегда ищет наидлиннейшее совпадение (longest match first).

Архитектура ядра замены (extensible для V2)
ReplacementEngine
├── interface ReplacementStrategy
│     └── process(textNode, rate, settings) → void
├── LetterReplacementStrategy   ← V1.0
│     ├── load mappings (latin + cyrillic)
│     ├── tokenize: longest-match scan
│     ├── randomly select rate/1000 of candidates
│     └── replace (text node или span если highlight)
└── WordTranslationStrategy     ← V2.0 hook (заглушка)
      └── fetch/lookup word → georgian equivalent
content.js создаёт ReplacementEngine, передаёт ему активную стратегию. Смена стратегии — замена одного объекта.

Алгоритм замены букв
dom-walker.js обходит DOM через TreeWalker — только текстовые узлы
Пропускаются теги: <script>, <style>, <code>, <pre>, <input>, <textarea>, <a> (URL не трогаем)
Для каждого текстового узла replacer.js:
Сканирует строку слева направо, ищет совпадения с таблицей (longest-match)
Строит список кандидатов на замену (все найденные совпадения)
Случайно выбирает rate/1000 долю кандидатов
Если highlight выключен: заменяет прямо в textContent
Если highlight включён: разбивает узел на части, заменённые оборачивает в <span class="geo-char" data-original="t" title="t">თ</span>
observer.js через MutationObserver отслеживает новые узлы и прогоняет их через тот же pipeline
Настройки (chrome.storage.sync)
{
  enabled: true,           // глобальный вкл/выкл
  rate: 100,               // замен на 1000 букв (0–1000)
  highlight: true,         // подсветка заменённых букв
  observeDynamic: true,    // MutationObserver
  activeMappings: ["latin-georgian", "cyrillic-georgian"]
}

Безопасность и стабильность
Нет eval, нет inline-скриптов — соответствует CSP MV3
Нет внешних запросов — таблицы бандлятся в расширение
Защита от повторной обработки — узлы помечаются атрибутом data-geo-processed
Дросселирование MutationObserver — батчинг через requestIdleCallback во избежание лагов
Skip editable elements — contenteditable, input, textarea никогда не трогаются

Скажи, можем ли мы это решить сейчас, это было бы хорошо. Или сложно, лучше запланировать на следующие версии:

1. Главная проблема — коллизии транслитерации. Например, в английском сочетание sh может быть как звуком «ш» (შ), так и двумя отдельными звуками на стыке морфем (например, в слове mishap), где замена на одну грузинскую букву дезориентирует. Еще сложнее с кириллицей: замена «х» на грузинскую ხ (твёрдая «х») или ჰ (мягкая «х») требует лингвистического анализа, а не простого Longest Match, иначе обучение будет ложным.

2. Отсутствие «Чёрного списка» и «Белого списка»: Необходим механизм исключения доменов (например, Google Docs или GitHub), где вмешательство в текст критически мешает работе. Давай добавим режимы белого и чёрного списка. Поле со списком доменов, по умолчанию в нём для примера гугл.докс (и может ещё какие-то сайты предложишь" и переключатель в режиме чёрного списка, чтобы не мешаать пользователю редактировать его документы и таблицы например. Надо будет - сам переключит и настроет.

3. Синхронизация: chrome.storage.sync имеет жесткие лимиты на объем данных (около 100 КБ); если таблицы соответствий или пользовательские словари (V2) вырастут, лимит будет быстро исчерпан. Нужно проверять, и не использовать много, зададим жёсткое ограничение, если можно - не более 10 кб. Запиши в планы.


Ещё обдумай такие возражения, есть ли в них вообще смысл и если да, то можем ли легко решить или запланируем на следующие версии:

1. Технические риски обработки домена: Алгоритм longest-match уязвим к краевым случаям (например, замена в составе URL, email или кода), а использование MutationObserver на <all_urls> без жёсткого дросселирования может вызвать заметные лаги на динамических сайтах вроде социальных сетей.

2. Нестабильность Service Worker: В Manifest V3 фоновый скрипт может быть завершён в любой момент, что создаёт риски потери состояния или прерывания обработки сообщений между контент-скриптом и интерфейсом расширения.

3. MutationObserver на динамических сайтах (Twitter, Facebook) вызовет тысячи срабатываний в секунду, замедляя загрузку страниц на 20–50% из-за батчинга и requestIdleCallback, несмотря на дросселирование. Алгоритм longest-match с приоритетами для каждого textNode на больших DOM-деревьях (например, новостные сайты) потребует избыточных вычислений, приводя к лагам и высоким затратам CPU >1%. Даже с data-geo-processed защита от повторной обработки не спасёт от частых мутаций в SPA-приложениях.

4. Vanilla JS без тестов на edge-кейсы (RTL-текст, emoji) усложнит отладку, а Manifest V3 с <all_urls> вызовет флаги безопасности в Chrome Web Store. V2 с переводами сломает extensibility: WordTranslationStrategy требует сетевых запросов, нарушая "no external requests" и CSP.

5. Редактируемый JSON в options.html уязвим к инъекциям, если валидация слаба — пользователь загрузит вредоносный mapping, вызвав DoS на DOM. host_permissions: ["<all_urls>"] мишень для abuse в store-аудите. - и это к вопросу о том, что пользователь не должен программировать

6. Использование MutationObserver на всех узлах («<all_urls>») — крайне ресурсоемкая задача для современного веба (бесконечные ленты соцсетей). Даже с requestIdleCallback, разбиение текстовых узлов на сотни <span> (при включенном highlight) кратно увеличивает количество DOM-узлов, что приведет к тормозам при скроллинге и «прыжкам» текста при рендеринге.

7. Метод TreeWalker может затронуть текст внутри кнопок, меню или иконок-шрифтов (например, Material Icons), где буква является техническим кодом символа. Замена одной буквы в слове-команде (например, в банковском приложении или админ-панели) может сделать интерфейс нечитаемым или визуально «развалить» верстку из-за разницы в высоте и ширине грузинских глифов.
